# Brand identity

Visual identity and voice guidelines for the 360 Flatmates platform.

## Brand colors

| Color | Hex | Usage |
|-------|-----|-------|
| Primary Blue | `#3B82F6` | Links, buttons, accents |
| Light Blue | `#60A5FA` | Hover states, highlights |
| Dark Background | `#0F172A` | Code blocks, dark mode |
| Gray | `#94A3B8` | Secondary text, borders |

## Typography

- **Headings**: System font stack (Helvetica Neue, Arial, sans-serif)
- **Body**: System font stack
- **Code**: Monospace (JetBrains Mono, Fira Code, monospace)

## Brand voice

### Principles

- **Clear**: Use plain language. Avoid jargon unless necessary.
- **Helpful**: Focus on what the user can do, not what the system does.
- **Direct**: Get to the point. Respect the reader's time.
- **Inclusive**: Write for a global audience. Avoid idioms and cultural references.

### Examples

**Good**: "Social AI Reply finds relevant Reddit posts and drafts helpful replies."
**Bad**: "Our cutting-edge AI-powered platform leverages advanced NLP to identify optimal engagement opportunities."

## Logo usage

The 360 Flatmates brand mark combines a circle (representing community) with an inner square (representing structure). Use the brand mark in:

- Documentation headers
- Video presentations
- Marketing materials

### Brand mark description

```
    ┌─────────┐
   ╱           ╲
  │  ┌───────┐  │
  │  │       │  │
  │  │   □   │  │
  │  │       │  │
  │  └───────┘  │
   ╲           ╱
    └─────────┘
```

The outer circle represents the global community of users. The inner square represents the structured, reliable platform. Together they symbolize "structured community engagement."

## Documentation style

### Headings

- Use sentence case for headings: "Getting started", not "Getting Started"
- Keep headings short and descriptive
- Use `##` for major sections, `###` for subsections

### Code blocks

- Always specify the language: ` ```python `, ` ```bash `
- Include comments for non-obvious code
- Keep code examples focused and runnable

### Links

- Use descriptive link text: [Getting started](getting-started.md), not [click here](getting-started.md)
- Link to related pages within the wiki
- Use absolute paths from repo root for source code references

### Tables

- Use tables for structured data (commands, configuration options, etc.)
- Keep tables focused on 3-5 columns maximum
- Include a header row
